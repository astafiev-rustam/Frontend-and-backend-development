## Введение
О системах контроля версий мы говорили еще в прошлом семестре. Концептуально, важно помнить, что в современных условиях системы контроля версий позволяют разработчикам отслеживать изменения в коде, работать над проектами в команде, а также откатывать изменения в случае необходимости. Git — это одна из самых популярных систем контроля версий, которая обеспечивает высокую скорость работы и гибкость.

По сути, программный продукт в такой системе контроля версий «собирается» из веток, каждая из которых представляет из себя определенный набор изменяемых компонентов приложения. Каждая из веток формируется за счет добавления коммитов, каждый из которых, фактически, представляет из себя контрольную точку сборки или версию определенной области редактируемого кода.

# Установка Git и среды разработки на компьютеры
## Установка на Windows
Скачайте установщик Git с официального сайта [1](https://git-scm.com/downloads/win).
Запустите установщик и следуйте инструкциям. Выберите параметры по умолчанию, если не уверены.
## Установка на macOS
Если у вас установлен Homebrew, просто выполните команду в терминале:
```bash
brew install git
```
Либо скачайте установщик с официального сайта [2](https://git-scm.com/downloads/mac).

## Установка на  Linux
Для Ubuntu или Debian выполните:
```bash
sudo apt update
sudo apt install git
```
Для Fedora:
```bash
sudo dnf install git
```
Для Arch Linux:
```
sudo pacman -S git
```
## Проверка установки
В конечном счете, после установки, в терминале необходимо выполнить команду:
git --version
В результате должна быть выведена версия Git, что и будет признаком верной установки.

# Основные команды Git
Подробно обо всех доступных командах для системы контроля версий можно почитать в официальной документации [4](https://git-scm.com/doc).
## Персонализация
Следующие команды настроят отображение имени пользователя, вносящего изменения для каждого коммита:
```
#Вместо <ваше_имя> можно ввести, например, astafiev_rustam
#Кавычки оставляем
git config --global user.name "<ваше_имя>"
#Теперь установим email. Принцип тот же.
git config --global user.email "<адрес_почты@email.com>"
```
## Создание репозитория
Для этого необходимо в терминале перейти в папку, в которой будет располагаться проект или запустить терминал из окружения этой папки и выполнить следующее:
```
#Для Linux и MacOS путь может выглядеть так /Users/UserName/Desktop/MyProject
#Для Windows например С://MyProject
cd <путь_к_вашему_проекту>
#Инициализация/создание репозитория
git init
```
## Добавление файлов
Для работы с файлами и кодом необходимо добавить файлы проекта в будущий коммит или по блокам в содержательную ветку. Для этого создадим 3 папки в нашем проекте «FromFront», «FromBack» и «KillerFicha».
Сейчас мы знаем, что последняя папка будет присутствовать, но пока притворимся, что нет. Добавить в первый коммит первые две папки и хранящиеся в них файлы (ПАПКИ НЕ ДОЛЖНЫ БЫТЬ ПУСТЫМИ). Для этого необходимо в терминале (в VS Code он доступен из папки проекта, как правило, в нижней части экрана) выполнить следующее:
```
git add FromBack
git add FromFront
```
## Создание коммита
Теперь может добавить коммит, в котором добавим в нашу ветку эти файлы:
```
git commit -m "<комментарий>"
```
Таким образом, любое создание, изменение или ошибки(?) можно фиксировать коммитами и фиксировать версии файлов в репозитории.
Естественно, не нужно фиксировать каждое изменение. Как правило, это делают при полноценном изменении некоторых блоков, добавлении функциональности или в конце рабочего дня.
## История изменений
Для просмотра истории коммитов можно использовать команду:
```
git log
```
# Работа с ветками
Ветки позволяют вам работать над разными функциями или исправлениями одновременно.
Создать новую ветку: `git branch <имя_ветки>`
Переключиться на другую ветку: `git checkout <имя_ветки>`
Чтобы объединить изменения из одной ветки в другую: `git merge <имя_ветки>`

Например, давайте смоделируем следующее:
1. Фронтендер начал работу над своими файлами в своей ветке.
2. Бэкендер начал работу над своими файлами в своей ветке.
3. Фронт закончил работу.
4. Изменения слили в одну ветку main.
5. Тимлид решил добавить киллер фичу в мэйн.
6. Бэк закончил работу.
7. Изменения слили в одну ветку main.
Все это далее представим только в виде команд. КРАЙНЕ рекомендуется действительно изменять файлы, чтобы далее можно было просматривать версии файлов и откатываться по ним.
## Фронтендер начал работу над своими файлами в своей ветке.
```
git branch Front
git checkout Front
git add FromFront
git commit -m "Создана ветка для работы фронтендера"
git checkout main
```
## Бэкендер начал работу над своими файлами в своей ветке.
```
git branch Back 
git checkout Back
git add FromBack  
git commit -m "Создана ветка для работы бэкендера" 
git checkout main
```
## Фронт закончил работу. 
```
git checkout Front
git add FromFront
git commit -m "Внесены ключевые изменения во фронтенд"
git checkout main
```
## Изменения слили в одну ветку main.
```
git merge Front
```
## Тимлид решил добавить киллер фичу в мэйн.
```
git add KillerFicha
git commit -m "Ребята, я тут такую штуку придумал"
```
## Бэк закончил работу.
```
git checkout Back
git add FromBack 
git commit -m "Внесены ключевые изменения в бэкенд" 
git checkout main
```
## Изменения слили в одну ветку main.
```
git merge Back
```
После выполнения всех команд можете самостоятельно посмотреть, какие изменения произошли и проверить количество и наличие коммитов в каждой ветке и в основной ветке через `git log`.

## Синхронизация локальных и удаленных репозиториев
Теперь попробуем опубликовать полученный репозиторий у себя в GitHub.
Для этого в панели работы с GitHub достаточно нажать на необходимый раздел и следовать указаниям. После внесения изменений в ветки их можно синхронизовать с удаленным репозиторием посредством команды `git push origin <имя_ветки>`.

Например, внесем изменения в один из файлов, закоммитим и синхронизуем ветки:
```
git add KillerFicha
git commit -m "Закончился рабочий день у Господина Тимлида"
git push origin main
```
Обратите внимание, что для добавления остальных веток необходимо их добавить отдельно.
Помимо этого, можно получать изменения из удаленного репозитория. Для этого необходимо использовать команду `git pull origin <имя_ветки>```.
Например, внесем изменения в один из файлов:
```
git pull origin main
```
Также в нем можно просматривать логи, возвращаться к версиям, вести статистику и прочее. Эта система не является единственной, но на практических занятиях будет использоваться она.
